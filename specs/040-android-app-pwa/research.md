# Research: Android App Distribution via Google Play

**Branch**: `040-android-app-pwa` | **Date**: 2026-03-08

## Unknown 1: Android Packaging Approach (TWA vs alternatives)

**Decision**: Trusted Web Activity (TWA) via Bubblewrap CLI.

**Rationale**:
- TWA is Google's official method for packaging a PWA as a Play Store app. It uses Chrome (or the system WebView on Android 10+) to render the PWA, meaning zero app logic needs to be rewritten.
- Preserves all PWA capabilities: Service Worker offline support, Web App Manifest, push notifications if added later.
- Full-screen mode with no browser chrome, exactly matching FR-002.
- No native code written — the Android project is a thin configuration shell generated by Bubblewrap CLI.

**Alternatives considered**:
| Alternative | Why Rejected |
|---|---|
| Capacitor | Adds a native JS bridge layer; heavier than needed since we have no native plugin requirements |
| Cordova | Legacy, deprecated, same overhead as Capacitor without modern tooling |
| Plain Android WebView | Requires managing WebView version independently; no TWA digital verification; historically poor PWA fidelity |
| React Native | Full app rewrite required — entirely separate codebase from PWA |

## Unknown 2: Bubblewrap CLI — Minimum Manifest Requirements

**Decision**: Current manifest is nearly TWA-ready with one required fix.

**Required manifest fields** (all present):
- `name` ✓ — "Graditone"
- `short_name` ✓ — "Graditone"
- `start_url` ✓ — set automatically by vite-plugin-pwa from `VITE_BASE`
- `display` ✓ — `"standalone"`
- `icons` ✓ — 192×192 and 512×512 PNGs present
- `background_color` ✓ — `#1a1a1a`
- `theme_color` ✓ — `#6366f1`

**Gap identified — icon `purpose` field**:  
Current manifest specifies `"purpose": "any maskable"` (space-separated) for both icons. Bubblewrap and Google Play require icons to declare `"maskable"` purpose separately from `"any"`. Fix: add two icon entries per size — one with `"purpose": "any"` and one with `"purpose": "maskable"`.

**Additional asset required for Play Store** (not in manifest):
- Feature graphic: 1024×500 PNG (mandatory for Play Store listing — not derivable from web manifest icons)
- Phone screenshots: minimum 2 screenshots at 1080×1920 (or 1080×2340 for taller phones)

**Bubblewrap version to use**: Latest stable (`@bubblewrap/cli@latest`, currently 1.22.x).

## Unknown 3: Digital Asset Links — Format and Hosting

**Decision**: Host `assetlinks.json` at `/.well-known/assetlinks.json` on the GitHub Pages domain, deployed as part of the existing PWA build.

**Rationale**: TWA requires Google to verify that the Android app is authorized to use the web domain. This is done via a Digital Asset Links JSON file served from the production URL. Without this file the TWA launches in a Chrome Custom Tab (with browser chrome visible), defeating FR-002.

**File location**: `frontend/public/.well-known/assetlinks.json`  
→ Deployed to: `https://graditone.com/.well-known/assetlinks.json`

**Expected format**:
```json
[{
  "relation": ["delegate_permission/common.handle_all_urls"],
  "target": {
    "namespace": "android_app",
    "package_name": "com.graditone.app",
    "sha256_cert_fingerprints": [
      "FILLED_IN_AFTER_KEYSTORE_GENERATION"
    ]
  }
}]
```

**Key constraint**: The SHA-256 fingerprint is derived from the app signing keystore. It must be generated once during setup and must not change across releases (changing it breaks all installed apps). If Google Play App Signing is used (recommended), two fingerprints must be listed: one for the upload key and one for the Play App Signing key.

**Serving note**: GitHub Pages serves static files. The `.well-known/` directory must be committed to `frontend/public/` and will be included automatically in the Vite build output and deployed by the existing `deploy-pwa.yml` workflow.

## Unknown 4: Firebase Crashlytics — PII Analysis

**Decision**: Firebase Crashlytics is compliant with the "no PII" requirement. Declare as "App diagnostics" in Play Store Data Safety form.

**What Crashlytics collects by default** (no SDK configuration needed to limit):
- Device model and manufacturer (e.g., "Samsung Galaxy S23")
- Android OS version
- App version (version name + version code)
- Crash stack trace and thread states
- A randomly generated installation ID (not tied to user identity; resets on app reinstall)

**What Crashlytics does NOT collect** unless explicitly configured:
- User identity, email, username
- Location (GPS or network-derived)
- Advertising ID
- Device serial number or IMEI

**Play Store Data Safety declaration**: Under "Data collected", select "App diagnostics" under the "Diagnostics" category. Mark as "Not linked to identity" and "Not shared with third parties" (Crashlytics is a Google service; for Play Store purposes data shared with Google's Firebase is considered first-party).

**Conclusion**: Consistent with Q1 ("no personal data") and FR-010. The distinction: "personal data" (PII) is not collected; anonymous diagnostic data (crash metadata) is collected and must be declared on the Data Safety form.

**Alternative considered**: Sentry (no Google dependency). Rejected because Firebase Crashlytics is free, zero-config, and simpler for a first Play Store release. Can be swapped later without affecting the spec.

## Unknown 5: Android Version Code Strategy

**Decision**: Derive `versionCode` from the npm `package.json` version using the formula `major × 10000 + minor × 100 + patch`. `versionName` equals the semantic version string.

**Rationale**:
- The existing CI already bumps the npm patch version on every main branch push.
- Using the same version for both the PWA and the Android app avoids diverged version numbers.
- The formula produces monotonically increasing integers (required by Play Store) for standard semantic versions up to 99.99.99.
- Example: `0.1.99` → `versionCode = 0×10000 + 1×100 + 99 = 199`

**Constraint**: `versionCode` must increase with every Play Store submission. The existing auto-bump on main guarantees this as long as `0.x.y` versions are used.

**Implementation**: In the `build-android.yml` CI workflow, read the version from `frontend/package.json` and compute `versionCode` before invoking Gradle.
