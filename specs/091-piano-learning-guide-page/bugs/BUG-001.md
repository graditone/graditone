# Bug Report: Learning Piano Guide Page Not Translated to Spanish

**Type**: Spec gap  
**Severity**: High  
**Feature**: `091-piano-learning-guide-page`  
**Reported**: 2026-05-20
**Status**: Patched  
**Patched**: 2026-05-20

## Description

The new "Learning Piano with Graditone" guide page is visible to users but **entirely untranslated in Spanish**. Every one of the 34 `guide.piano.*` keys in `frontend/src/i18n/locales/es.json` contains a stub value — the original English text prefixed with `[ES]` (e.g., `"[ES] Learning Piano with Graditone"`). Spanish-speaking users will see the raw `[ES]` prefix and English content instead of a localized experience.

**Affected file**: `frontend/src/i18n/locales/es.json`  
**Affected key namespace**: all 34 `guide.piano.*` keys  
**Example**:
```json
"guide.piano.page_title": "[ES] Learning Piano with Graditone"
```
**Expected**:
```json
"guide.piano.page_title": "Aprende Piano con Graditone"
```

## Artifact Traceability

### spec.md

- **Affected user story**: User Story 1 — Discover How Graditone Supports Piano Practice (P1); indirectly all four user stories because all rendered text is untranslated.
- **Affected requirements**:
  - **FR-010**: "All text on the guide page MUST use the existing i18n infrastructure so that content strings are ready for translation." — FR-010 was satisfied mechanically (all text uses `t()` calls), but it never required that a Spanish translation be *delivered*. This is the gap.
- **Gap identified**: The spec has no requirement that the Spanish locale (`es.json`) contain actual translations for the new `guide.piano.*` keys. FR-010 only mandates i18n-readiness ("ready for translation"), not delivery of translations. A requirement such as "The guide page MUST be available in all supported app languages (English and Spanish) at launch" is absent. The edge-case note "All guide content should be authored using the existing i18n infrastructure so translations can be added *incrementally*" explicitly permits deferred translation, which enabled the stub approach to be accepted as complete.
- **Affected success criterion**: SC-007 — "All i18n strings used on the guide page are defined in the existing translation files with at least English content; no hardcoded strings outside i18n." SC-007 only requires English content and is silent about Spanish parity.

### plan.md

- **Affected sections**:
  - **Phase 0 – Key decisions** (`i18n key namespace`): States keys will be added to both `en.json` and `es.json` but describes ES entries as "stubs" with `[ES]` prefix, deferring translation indefinitely.
  - **Phase 1 – i18n Contract**: Explicitly documents the stub strategy: *"ES stubs = EN text with `[ES]` prefix until translated."* No follow-up action or task is scheduled to replace stubs with real translations.
  - **Project Structure**: Lists `es.json` as `MODIFIED` but scopes that modification only to adding stubs.
- **Impact**: The plan must be updated to include a new section or note in Phase 7 (Polish & Cross-Cutting Concerns) explicitly requiring Spanish translation delivery before the feature is considered production-ready.

### tasks.md

- **Affected tasks**:
  - **T002** `[X]` — "Add matching `guide.piano.*` stub keys to `frontend/src/i18n/locales/es.json` (copy EN text; prefix each value with `[ES]` pending translation)" — marked **complete**, but adding stubs is not the same as delivering the Spanish translation. The task definition itself scoped completion to stub creation, causing the translation work to fall through the cracks.
- **False completions**: T002 was accepted as done when stubs were added. If "done" means the feature is ready to ship to Spanish-speaking users, T002 was completed prematurely.
- **Missing tasks**:
  - A new task is needed: **T-NEW**: "Translate all 30 `guide.piano.*` keys in `es.json` from English to Spanish — replace every `[ES]` stub with proper Spanish text, following the tone and vocabulary of the existing translated keys in `es.json`."

## Root Cause Analysis

The root cause is a **spec gap** compounded by an **accepted-as-complete task scope that was too narrow**:

1. `spec.md` FR-010 required i18n infrastructure readiness but never mandated Spanish translation delivery. The phrase "ready for translation" was interpreted as a technical/structural requirement (use `t()` calls) rather than a content requirement (provide translated strings).
2. The plan deliberately adopted a stub-first strategy (`[ES]` prefix) as an explicitly deferred placeholder, but no task or spec requirement was ever created to close that deferral.
3. T002 was scoped to stub creation only. Once stubs were committed, the task was marked `[X]` with no follow-up work item tracking the remaining translation debt.
4. SC-007 (the success criterion closest to this issue) only checks for English content and is silent on language parity, so the feature passed its own success criteria without providing Spanish translations.
