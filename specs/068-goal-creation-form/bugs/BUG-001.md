# Bug Report: User-Uploaded Score Always Blocked on Goal Creation

**Type**: Implementation drift  
**Severity**: High  
**Feature**: `068-goal-creation-form`  
**Reported**: 2026-05-20  
**Status**: Patched  
**Patched**: 2026-05-20

## Description

When a user tries to create a goal for a **user-uploaded score** (a score loaded via file upload, not a preloaded catalogue score), the Goal Creation Form incorrectly marks the score as "unavailable." The submit button is permanently disabled and a warning "Score no longer available — please select a different score" is shown, even though the score was just selected and is fully accessible.

**Steps to reproduce**:
1. Upload a score (e.g., via the file picker in the app).
2. Open the Sessions plugin → Goals tab.
3. Tap "Create Goal."
4. In the creation form, open the score picker and select the user-uploaded score.
5. Observe: the score selector shows the selected score's title, but a warning icon appears immediately and the submit button remains disabled.

**Expected**: The form accepts the user-uploaded score and enables the submit button, allowing goal creation.  
**Actual**: The `scoreUnavailable` flag is permanently `true` for all user-uploaded scores, disabling submission.

  - **FR-011**: "The form MUST block submission if no score is selected or **if the selected score is unavailable**." The intent of "unavailable" is a score that was previously selected but has since been removed, not a user-uploaded score that is actively present.
- **Gap identified**: FR-011 does not define what "unavailable" means for user scores (scores with `type: 'user'`). It only documents the preloaded-score unavailability scenario (score removed from catalogue after selection). This ambiguity allowed the implementation to apply a catalogue-only check to all score types.

## Root Cause Analysis

In `GoalCreationForm.tsx`, the `scoreUnavailable` check is:

```typescript
const catalogue = context.scorePlayer.getCatalogue();
const scoreUnavailable = selectedScore !== null &&
  !catalogue.some(c => c.id === selectedScore.ref.id);
```

`context.scorePlayer.getCatalogue()` returns only **preloaded** catalogue scores. User-uploaded scores have `ref.type === 'user'` and a storage-generated UUID as their `id` — neither of which will ever appear in the preloaded catalogue. As a result, any user-uploaded score selected via `handleSelectUserScore` will always evaluate `scoreUnavailable = true`, permanently disabling the submit button.

The correct pattern — used in `TaskRow.tsx` — gates the catalogue check on the score type:
```typescript
const scoreUnavailable = task.scoreRef.type === 'preloaded' && !catalogue.some(c => c.id === task.scoreRef.id);
```

The same guard was not applied in `GoalCreationForm.tsx` during implementation of T016.

This is a **spec ambiguity made concrete as implementation drift**: FR-011 only described the preloaded-score removal scenario but didn't explicitly exclude user scores from the availability check. The implementation then applied the most literal reading of "check against catalogue" without the type guard.

## Recommended Fix

1. Run **`/speckit.bugfix.patch`** to:
   - Add the type guard to `GoalCreationForm.tsx`:
     ```typescript
     const scoreUnavailable = selectedScore !== null &&
       selectedScore.ref.type === 'preloaded' &&
       !catalogue.some(c => c.id === selectedScore.ref.id);
     ```
   - Update FR-011 in `spec.md` to clarify that unavailability only applies to preloaded catalogue scores.
   - Reopen T016 in `tasks.md` and add a regression test task for user-uploaded score goal creation.
2. Run **`/speckit.bugfix.verify`** to confirm consistency across all spec artifacts.
3. Resume **`/speckit.implement`** to apply the code fix and add the regression test.
